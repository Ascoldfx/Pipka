#backup

# Бэкапы БД

Файл: `app/services/backup_service.py`. Запускается из [[Сервисы#scheduler]] cron-job'ом `daily_backup` в **02:30 UTC** ежедневно.

## Поток

```
run_backup()
   ├─ если DATABASE_URL не PostgreSQL → return ""  (skip, dev sqlite)
   ├─ создаём /app/data/backups/  (volume app_data)
   ├─ pg_dump -h db -p 5432 -U pipka pipka  (PGPASSWORD через env)
   ├─ gzip → pipka_YYYYMMDD_HHMMSS.sql.gz
   ├─ ротация: оставляем последние 7 файлов
   └─ если B2_KEY_ID/B2_APP_KEY/B2_BUCKET заданы → upload в Backblaze (best-effort)
```

## Локальный шаг

`pg_dump` запускается через `subprocess.run` (sync, в основном thread; OK потому что 35MB БД дампится за <1с). Пароль передаётся через переменную окружения `PGPASSWORD`, чтобы не светиться в `ps`. Stdout пайпится напрямую в `gzip.open(...)` — не кладём промежуточный незажатый дамп на диск.

Файлы: `/app/data/backups/pipka_YYYYMMDD_HHMMSS.sql.gz`. Volume `app_data` примонтирован к контейнеру (см. [[Деплой]]).

Ротация: `_rotate_backups()` сортирует файлы по имени (timestamp в имени гарантирует хронологический порядок) и удаляет всё кроме последних `KEEP_LAST = 7` файлов.

## Backblaze B2 (опционально)

`_upload_b2(path)` — best-effort, через `boto3` S3-совместимый клиент. Endpoint: `https://s3.us-west-004.backblazeb2.com` (`B2_ENDPOINT`, можно переопределить).

Ключ объекта: `db-backups/pipka_YYYYMMDD_HHMMSS.sql.gz`. Заливается через `loop.run_in_executor` чтобы не блокировать event loop.

**Важно:** падение B2 НЕ роняет основной бэкап — локальный файл уже сохранён и виден на диске. В логах будет `WARNING: B2 upload failed (local backup is intact): <error>`.

Если ключи не заданы (любой из `B2_KEY_ID`/`B2_APP_KEY`/`B2_BUCKET` пустой) — upload просто пропускается.

## Восстановление

```bash
ssh root@217.76.61.28 -i ~/.ssh/id_ed25519
docker exec pipka-app-1 ls -la /app/data/backups/
# выбрать файл, например pipka_20260427_023000.sql.gz

# Распаковать и залить в текущую БД (ВНИМАНИЕ: сначала pg_drop tables!)
docker exec pipka-app-1 sh -c \
  'gunzip -c /app/data/backups/pipka_20260427_023000.sql.gz | \
   PGPASSWORD=$POSTGRES_PASSWORD psql -h db -U pipka pipka'
```

Из B2:

```bash
aws s3 cp \
  --endpoint-url https://s3.us-west-004.backblazeb2.com \
  s3://pipka-backups/db-backups/pipka_YYYYMMDD_HHMMSS.sql.gz ./
```

(Использовать `aws configure` с B2 ключами, либо `boto3` напрямую.)

## Проверка целостности

Сейчас автоматической проверки восстановимости нет — пункт из [[Roadmap]]. Раз в месяц рекомендую вручную:

1. Скачать последний `*.sql.gz` (с B2 или локально).
2. `gunzip -c file.sql.gz | psql --schema-only -d throwaway_db` — убедиться, что схема грузится.

## Связи

- Cron-job `daily_backup` зашедулен в [[Сервисы#scheduler]] на 02:30 UTC.
- Зависимости: `postgresql-client` в Dockerfile (для `pg_dump`), `boto3>=1.34` в pyproject (только для B2 upload).
- Конфиг: `B2_KEY_ID`, `B2_APP_KEY`, `B2_BUCKET`, `B2_ENDPOINT` — см. [[Настройки#бэкапы-бд-backblaze-b2-опционально]].
