# Кэш скоринга и инвалидация

С Phase 2 каждая `JobScore` строка несёт два поля провенанса (см. [[База данных#job_scores]]):

- `profile_hash: str(64) NULL` — sha256 от 10 scoring-relevant полей `UserProfile`.
- `model_version: str(64) NULL` — `"gemini:gemini-3.1-flash-lite-preview"`, `"claude:claude-sonnet-4-20250514"`, `"nvidia:google/gemma-4-31b-it"`, или `"prefilter"`.

Composite-индекс `ix_job_scores_user_profile_model (user_id, profile_hash, model_version)` обслуживает lookup'ы.

## Помогает решить

1. **Профиль изменился** → старые AI-оценки устарели, потому что промпт строился на других `target_titles`/`languages`/`excluded_keywords` etc.
2. **AI-модель сменилась** (env-апгрейд `GEMINI_MODEL`) → результаты другой модели не должны сравниваться напрямую со старыми.
3. **Распознавание pre-filter rejects** → отличить `score=0` от rule-based фильтра от `score=0` от AI ("AI confirmed: not a fit").

## profile_hash

Файл: `app/scoring/profile_hash.py`. Функция `compute_profile_hash(profile)`:

- Берёт 10 полей: `resume_text`, `target_titles`, `languages`, `work_mode`, `preferred_countries`, `excluded_keywords`, `english_only`, `target_companies`, `min_salary`, `experience_years`.
- Whitespace-стрипает строки, сортирует списки. Значит `"Sales Manager " → "Sales Manager"` и `["A", "B"] vs ["B", "A"]` дают одинаковый hash — тривиальные правки не инвалидируют кэш.
- Сериализует в стабильный JSON (`sort_keys=True`), хеширует sha256.

`MODEL_GEMINI()`, `MODEL_CLAUDE()`, `MODEL_NVIDIA()` — фабрики, читают `settings.{gemini,claude,nvidia}_model` динамически. Смена модели через env (без передеплоя кода) автоматически инвалидирует кэш.

## Read path (Phase 2b)

В [[Сервисы#scheduler|_score_and_notify]], [[Сервисы#scheduler|_backfill_score]] и `score_jobs` (matcher.py cache):

```sql
WHERE user_id = X
  AND (profile_hash IS NULL                  -- legacy, до Phase 2a
       OR profile_hash = current_hash)
```

| Состояние строки | Поведение |
|------------------|-----------|
| `profile_hash IS NULL` | Считается "уже оценено" (legacy compat). Никогда не пере-оценивается. |
| `profile_hash = current_hash` | Cache hit. Пропускаем. |
| `profile_hash != current_hash` | Не в множестве "оценено" → попадает в очередь backfill'а. |

## Write path — UPSERT

Все JobScore-INSERT'ы идут через `pg_insert(...).on_conflict_do_update(...)`:

```python
stmt.on_conflict_do_update(
    index_elements=["job_id", "user_id"],
    set_={
        "score": stmt.excluded.score,
        "ai_analysis": stmt.excluded.ai_analysis,
        "scored_at": datetime.now(),
        "profile_hash": stmt.excluded.profile_hash,
        "model_version": stmt.excluded.model_version,
    },
    where=JobScore.profile_hash != stmt.excluded.profile_hash,
)
```

`NULL != X` в Postgres — unknown, не TRUE → legacy NULL-строки UPDATE'ом не задеваются. Свежий скоринг с тем же hash — no-op (не туда дёрнем `scored_at`). Mismatched-строки — перезаписываются.

## Эффект на квоты

При смене профиля стартует **постепенное** освежение, не штормовое:

- 2 backend'а × 1000 jobs/run × 12 раз/сутки (`_backfill_score` каждые 2ч) = ~24k job-scoring/сутки.
- Реалистично user'ская база ~7000 scored — освежается за <1 суток.
- Real-time `_score_and_notify` обновляет mismatched на ходу при следующем скане.

Никакого глобального DELETE/UPDATE на endpoint'е смены профиля. AI-квоту не сжигаем за раз.

## prefilter sentinel

Pre-filter rejects (см. [[Сервисы#scheduler|_backfill_score]]) пишутся с `model_version="prefilter"`. Это позволяет:

- `recheck_zero_scores` (см. [[Скоринг#recheck]]) находить именно rule-based нули (`score=0 AND ai_analysis IS NULL`) и пере-проверять их AI-моделью на ложноотрицательные срабатывания pre-filter'а.
- `excluded_keywords` change тоже триггерит инвалидацию prefilter-rejects: profile_hash меняется → row mismatched → попадает в очередь, заново пропускается через `pre_filter()`, либо проходит, либо опять отвергается с новым hash.

## Что НЕ делает Phase 2

- Не удаляет ничего. Историческая `scored_at` сохраняется.
- Не двигает legacy NULL-строки. Их 7700 — они frozen.
- Не блокирует apply/reject. Эти действия идут через [[Трекер]], никак не зависят от кэша скоринга.
