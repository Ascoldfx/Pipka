# Миграции БД (Alembic)

Schema-as-code. Все DDL-изменения — только через Alembic. Прямой `Base.metadata.create_all()` в runtime запрещён (правило в `CLAUDE.md`).

## Структура

```
alembic/
├── env.py           — async engine, импортирует Base.metadata
├── script.py.mako   — шаблон новой миграции
└── versions/
    ├── 0001_baseline.py
    ├── 0002_phase2_profile_hash.py
    ├── 0003_job_url_status.py
    └── 0004_search_embeddings.py
alembic.ini
```

## Lifecycle на проде

При старте контейнера `app/database.py:init_db()`:

```python
async def init_db():
    def _run_alembic_upgrade():
        cfg = Config(str(repo_root / "alembic.ini"))
        cfg.set_main_option("sqlalchemy.url", settings.database_url)
        command.upgrade(cfg, "head")

    await asyncio.get_running_loop().run_in_executor(None, _run_alembic_upgrade)
```

`run_in_executor` — потому что Alembic sync, а startup async. Запускает в thread-pool worker, не блокирует event loop.

При первой загрузке без `alembic_version` table → создаётся, применяются все миграции от `None` до `head`. На уже-провизионированной БД `0001_baseline` идемпотентен (см. ниже), `0002+` применяются как обычно.

## Существующие миграции

### 0001_baseline.py

```python
def upgrade():
    from app.models import Base
    Base.metadata.create_all(bind=op.get_bind())

def downgrade():
    raise RuntimeError("Refusing to downgrade past the baseline migration.")
```

Идемпотентен: `create_all` пропускает существующие таблицы. На fresh dev DB разворачивает всю схему. На проде (где все 8 таблиц + индексы созданы исторически) — no-op.

`downgrade` сознательно рейзит, чтобы случайный `alembic downgrade base` не дропнул всю продовую БД.

### 0002_phase2_profile_hash.py

```python
def upgrade():
    op.execute("ALTER TABLE job_scores ADD COLUMN IF NOT EXISTS profile_hash VARCHAR(64)")
    op.execute("ALTER TABLE job_scores ADD COLUMN IF NOT EXISTS model_version VARCHAR(64)")
    op.execute("""
        CREATE INDEX IF NOT EXISTS ix_job_scores_user_profile_model
        ON job_scores (user_id, profile_hash, model_version)
    """)
```

Добавляет fingerprint-колонки для [[Кэш и инвалидация]]. Использует `IF NOT EXISTS` чтобы быть идемпотентной на проде, где колонки могли быть добавлены вручную через ALTER до bootstrap'а Alembic.

### 0003_job_url_status.py

Добавляет `jobs.url_status` (`'active'|'closed'|'unreachable'|NULL`), `jobs.url_checked_at`, `jobs.url_check_failures`, плюс composite-индекс `ix_jobs_url_status_checked (url_status, url_checked_at)` для picker'а ([[Проверка ссылок]]) и фильтра "скрыть закрытые" в `/api/jobs`.

### 0004_search_embeddings.py

PostgreSQL-only. Добавляет:
- `CREATE EXTENSION IF NOT EXISTS vector` — pgvector (доступен из `pgvector/pgvector:pg16` image, см. [[Деплой]]).
- `jobs.search_vector` — generated `tsvector` поверх title (вес A) + company_name (A) + description (B), с GIN-индексом `ix_jobs_search_vector`. Питает `?search=…` через `websearch_to_tsquery('simple', term)`.
- `jobs.embedding vector(768)`, `jobs.embedding_model`, `jobs.embedding_updated_at` + аналогичные колонки в `user_profiles`. HNSW cosine-индексы `ix_jobs_embedding_hnsw` / `ix_profiles_embedding_hnsw` — для ANN nearest-neighbour поиска ([[Поиск и индексация]]).

На SQLite миграция no-op (там нет ни tsvector, ни pgvector).

## Создание новой миграции

```bash
# Локально, на свежем sqlite (или против stage Postgres)
DATABASE_URL=sqlite+aiosqlite:///./_test.db alembic revision \
    --autogenerate -m "<описание>"
```

Autogenerate сравнивает `Base.metadata` (из `app.models`) с DB и пишет diff в новый файл `alembic/versions/000N_<описание>.py`. **Обязательно проверить вручную** — autogenerate не ловит:

- GIN-индексы (expression indexes).
- Раздельные NULL/NOT NULL transitions.
- Type changes (json → jsonb): нужно `op.alter_column("...", type_=JSONB(), postgresql_using="...::jsonb")`.

После проверки:

1. Закоммитить файл (см. [[#gitignore-trap]]).
2. Запушить.
3. На проде `git pull && docker compose up -d --build`.
4. На старте контейнера `init_db` сам применит.

Можно вручную: `docker exec pipka-app-1 alembic upgrade head`.

## Concurrent index creation

Для горячих таблиц (jobs ~7700, job_scores ~7700) `op.create_index` блокирует таблицу на время построения. Для production-grade миграций используем raw SQL:

```python
def upgrade():
    op.execute("CREATE INDEX CONCURRENTLY ix_jobs_xyz ON jobs (xyz);")

def downgrade():
    op.execute("DROP INDEX CONCURRENTLY ix_jobs_xyz;")
```

Но `CONCURRENTLY` нельзя в transaction → нужно `with op.get_context().autocommit_block():`. Альтернатива — применить индекс вручную SQL'ем на проде (как было сделано для `ix_jobs_scraped_at`, `ix_jobs_country_scraped`, `ix_job_scores_user_scored_at`, `ix_jobs_merged_sources`), а Alembic-миграция использует `CREATE INDEX IF NOT EXISTS`.
