# Дедупликация вакансий

За один скан 9 источников возвращают ~2500–3000 raw-объектов. Часть из них — одна и та же вакансия с разных площадок (например, "Heraeus" на LinkedIn и "Heraeus Quarzglas GmbH & Co. KG HRdirekt" на Indeed). Pipeline дедупа в `app/sources/aggregator.py` сжимает это в ~1500 уникальных и обогащает `merged_sources`.

Два прохода: **exact** → **fuzzy**.

## Pass 1 — exact: dedup_hash

Файл: `app/sources/base.py:RawJob.dedup_hash` — sha256 от нормализованной пары `(title, company)`. Считается на этапе парсинга источника. На уровне БД — `UNIQUE INDEX ix_jobs_dedup_hash`.

В aggregator:

```python
seen_hashes: set[str] = set()
unique = []
for job in raw_jobs:
    if job.dedup_hash not in seen_hashes:
        seen_hashes.add(job.dedup_hash)
        unique.append(job)
```

Этот шаг ловит точные дубликаты — одна и та же вакансия, попавшая дважды от одного источника, или из разных источников но с идентичным title+company.

Типичная редукция: **2700 → 2200 вакансий** (~18%).

## Pass 2 — fuzzy: title + company subset

Файл: `app/sources/base.py:is_fuzzy_duplicate(a, b)`. O(n²) по `unique`-списку, на n<2500 это <1с в Python.

Критерии слияния:

1. **Title match** — после нормализации (lower, удаление пунктуации, стоп-слова "senior/director/the/an") заголовки совпадают.
2. **Company subset** — frozenset токенов длиной ≥4 символов одного — подмножество токенов другого. Пример: `{heraeus} ⊆ {heraeus, quarzglas, gmbh, hrdirekt}` → match. Fallback для коротких названий (BMW): прямое равенство.
3. **Location guard** — `_locations_conflict()`: разные города → НЕ объединять (Berlin ≠ München, даже если title и company совпадают). Асимметричный случай ("Berlin" vs `None`) — НЕ конфликт, объединяем.

Если совпало:

```python
all_sources = seen.raw_data.get("merged_sources", [seen.source])
if job.source not in all_sources:
    all_sources.append(job.source)

# Выбираем "канонический" вариант — у кого описание длиннее
if len(job.description or "") > len(seen.description or ""):
    job.raw_data["merged_sources"] = all_sources
    fuzzy_deduped[i] = job
else:
    seen.raw_data["merged_sources"] = all_sources
```

Логика приоритизации описания: чем длиннее — тем больше шансов, что pre-filter и AI-скоринг получат полный контекст. См. [[Скоринг]].

Типичная редукция Pass 2: **2200 → 2150** (~2%).

## Результат: merged_sources

`Job.raw_data["merged_sources"]` — `list[str]` источников, которые видели эту вакансию. Используется:

- В UI карточки — золотой бэдж `⊕ слито N` показывает, что вакансия пришла из нескольких источников.
- В [[Ops панель|/api/ops/dedup]] — отдельный список для админа: вакансии с `merged_sources.length > 1`. Бэкенд: `SELECT ... WHERE jsonb_array_length(raw_data->'merged_sources') > 1`. Ускорено GIN-индексом `ix_jobs_merged_sources` на `(raw_data->'merged_sources')` (создан в [[Миграции|0001_baseline]] через ALTER на проде).
- В фильтре источников — Watchlist-вакансии получают virtual source `watchlist`, остальные источники остаются настоящими.

## Hidden dedup

Помимо агрегатора, [[Трекер]] использует `Job.dedup_hash` для скрытия "уже-скрытых" вакансий: если пользователь reject'ил вакансию X, и завтра приходит ту же X из другого источника (новый `Job.id`, тот же `dedup_hash`) — `get_hidden_dedup_hashes(user_id)` подсунет её в скип-лист скана и AI её не увидит.
