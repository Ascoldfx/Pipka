# Поиск и индексация

Слой retrieval перед AI-скорингом. Цель — быстро находить релевантные вакансии в базе и постепенно уменьшать количество дорогих LLM-вызовов.

## Full-text search

Миграция: `alembic/versions/0004_search_embeddings.py`.

В `jobs` добавлена generated-колонка `search_vector`:

- `title` — вес `A`
- `company_name` — вес `A`
- `description` — вес `B`

Индекс:

```sql
CREATE INDEX ix_jobs_search_vector
ON jobs USING gin (search_vector);
```

`GET /api/jobs?search=...` на PostgreSQL использует:

```sql
search_vector @@ websearch_to_tsquery('simple', :term)
```

Для SQLite/dev остаётся fallback через `ILIKE`.

## Semantic indexing

Миграция включает `pgvector`:

```sql
CREATE EXTENSION IF NOT EXISTS vector;
```

Docker DB image переключён на `pgvector/pgvector:pg16`, иначе стандартный `postgres:16-alpine` не содержит extension `vector`.

Колонки:

- `jobs.embedding vector(768)`
- `jobs.embedding_model`
- `jobs.embedding_updated_at`
- `user_profiles.embedding vector(768)`
- `user_profiles.embedding_model`
- `user_profiles.embedding_updated_at`
- `user_profiles.embedding_profile_hash`

Индексы:

- `ix_jobs_embedding_hnsw`
- `ix_user_profiles_embedding_hnsw`

Оба используют cosine distance:

```sql
embedding vector_cosine_ops
```

## Embedding backend

Файл: `app/services/embedding_service.py`.

Backend: Google Gemini Embedding через `google-generativeai`.

Настройки:

| Env | Default |
|-----|---------|
| `EMBEDDING_ENABLED` | `true` |
| `EMBEDDING_MODEL` | `models/gemini-embedding-001` |
| `EMBEDDING_DIMENSION` | `768` |
| `EMBEDDING_JOBS_PER_RUN` | `70` |
| `EMBEDDING_PROFILES_PER_RUN` | `20` |
| `EMBEDDING_INDEX_INTERVAL_HOURS` | `2` |
| `SEMANTIC_SEARCH_LIMIT` | `500` |

Если нет `GEMINI_API_KEY`, сервис no-op.

## Scheduler

Job: `embed_index`.

Расписание: каждые `EMBEDDING_INDEX_INTERVAL_HOURS`, плюс startup run через 90 секунд.

Что делает:

1. Берёт новые вакансии без `jobs.embedding`.
2. Строит текст: title, company, location, country, description.
3. Получает embedding с task type `retrieval_document`.
4. Обновляет `jobs.embedding`.
5. Берёт профили, у которых `embedding_profile_hash` не совпадает с текущим `profile_hash`.
6. Получает embedding профиля с task type `retrieval_query`.
7. Обновляет `user_profiles.embedding`.

События пишутся в `ops_events` как `embedding_index`.

## Semantic API

`GET /api/jobs?semantic=1` ограничивает выборку top-N вакансиями по близости к embedding профиля:

```sql
ORDER BY jobs.embedding <=> user_profiles.embedding
```

`include_closed=0` продолжает скрывать вакансии с `url_status='closed'`.

## Что это даёт

- Быстрый текстовый поиск без `ILIKE '%term%'` по всей таблице.
- Персональный semantic retrieval по резюме/профилю.
- Основа для следующего шага: передавать в Gemini/Claude/NVIDIA только top-N кандидатов после vector pre-rank.
