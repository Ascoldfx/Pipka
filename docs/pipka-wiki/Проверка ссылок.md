# Проверка ссылок (URL liveness)

Ежесуточный HEAD-пинг по `Job.url` чтобы вычистить из инбокса вакансии, страница которых уже снята с сайта-источника. До этой фичи `_cleanup_old_jobs` удалял вакансии только по возрасту (>45 дней) — закрытая через неделю болталась в выдаче месяц.

Файлы: `app/services/url_checker.py` (логика), `app/services/scheduler_service.py:_check_job_urls` (вход), миграция `alembic/versions/0003_job_url_status.py` (схема), `app/api/jobs.py` (фильтр).

## Состояния `Job.url_status`

| Значение | Когда | Поведение в UI |
|----------|-------|----------------|
| `NULL` | Вакансия ещё не проверялась (новенькая или до запуска фичи) | Показывается как обычно |
| `active` | HEAD вернул 2xx | Показывается как обычно |
| `closed` | HEAD вернул 404/410, или 3xx → редирект на листинг (см. `_classify_redirect`) | Скрыта из инбокса (`include_closed=0` дефолт), при показе — line-through + бейдж `🚫 закрыта` |
| `unreachable` | После `url_check_max_failures=3` подряд транзиентных ошибок (5xx, timeout, network error) | Показывается, но с предупреждающим бейджем `⚠ недоступна` |

`NULL` намеренно НЕ считается closed — мы не хотим прятать только что собранные вакансии до их первой проверки.

## Pipeline

```
APScheduler @ 04:00 UTC → _check_job_urls()
   │
   ├─► run_url_check_pass(session)
   │     ├─ SELECT 500 oldest-checked jobs (NULL first)
   │     ├─ Semaphore(10) global, async lock per host
   │     ├─ HEAD with User-Agent: "Pipka-Liveness/1.0 (+https://pipka.net)"
   │     │     timeout 10s, follow_redirects=False
   │     │
   │     ├─ classify():
   │     │     2xx        → active
   │     │     404/410    → closed
   │     │     3xx        → _classify_redirect(Location)
   │     │     401/403    → active   (auth wall, страница есть)
   │     │     405        → fallback to GET, повтор классификации
   │     │     5xx/network → unreachable, transient
   │     │
   │     └─ commit: url_status, url_checked_at, url_check_failures
   │
   └─► OpsEvent("url_check", "success") c counters в payload
```

`_classify_redirect`: если `Location` ведёт на корень / `/jobs` / `/careers` / `/search` без числового или 8+-значного токена в пути — считаем "redirect to listing" и помечаем как closed. Это типичный паттерн снятой вакансии у LinkedIn/Indeed/Xing.

## Throttling per host

`_pace_host(host)` — async-lock + monotonic pacer. Между двумя запросами к одному хосту минимум `url_check_per_host_delay = 1.5s`. Это критично для агрессивных к ботам сайтов (LinkedIn captcha при 20+ rps).

Глобально — `Semaphore(url_check_concurrency = 10)`: 10 разных хостов могут долбиться параллельно, на одном хосте — последовательно.

## Соответствие квоте

Daily 500 jobs × 1 HEAD = 500 запросов сутки. На 7700 вакансий полный цикл проверки — **15 дней**. Достаточно: вакансия закрылась → попадает в очередь проверки → через ≤24ч помечается closed → исчезает из инбокса.

## API

`GET /api/jobs?include_closed=0` (default) — фильтрует строки с `url_status='closed'`. `=1` — показывает все. `unreachable` всегда видим (это soft-signal, а не подтверждённая закрытость).

В JSON-ответе `/api/jobs` появляются:
- `url_status: 'active'|'closed'|'unreachable'|null`
- `url_checked_at: ISO timestamp | null`

## UI

В тулбаре фильтров — чекбокс "Показывать закрытые" (RU/EN/DE/ES). По умолчанию выкл.

В таблице вакансий: при `url_status='closed'` строка получает opacity 0.55 + line-through на title + красный бейдж `🚫 закрыта` рядом с заголовком. При `url_status='unreachable'` — жёлтый бейдж `⚠ недоступна` без затемнения.

## Ops визивизионость

`OpsEvent("url_check", "success")` пишется после каждого тика с payload:

```json
{"checked":500, "active":312, "closed":47, "unreachable":141, "skipped":0}
```

См. [[Ops панель#ops_events]] — добавлен в каталог event_types.

## Soft-404 (откладывается)

Многие сайты возвращают HTTP 200 с body "this position has been filled" вместо честного 404. Per-source маркеры (`SOFT_404_MARKERS = {"linkedin": ["this job is no longer accepting applications"], ...}`) — пункт [[Roadmap]]. Сейчас: только HEAD-пинг.
